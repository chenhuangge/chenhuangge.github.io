软件测试自学基础

Mar 12, 2019
3 min read

  1. 测试目标 : 测出系统软件缺陷，尽可能的早一些，并确保其得以修复，从而评估系统的功能、性能，指导下一步计划

  2. 软件缺陷的定义 软件未达到产品说明书标明的功能 软件出现了产品说明书指明不会出现的错误 软件功能超出产品说明书指明范围 软件未达到产品说明书虽未指出但应到达的目标 软件测试人员认为软件难以理解、不易使用、运行速度缓慢，或者最终用户认为不好

  3. 测试需求分析

可能有些人认为测试需求分析无关紧要，这种想法是很不对的。需求分析不但重要，而且至关重要 功能测试需求分析 测试环境需求分析 测试资源需求分析

  1. 测试计划

测试背景：软件项目介绍、项目涉及人员及联系方式 测试依据：软件需求文档、软件规格书、软件设计文档 测试资源：测试设备需求、测试人员需求、测试环境需求 测试策略：采取测试方法、搭建哪些测试环境、采取哪些测试工具以测试管理工具、对测试人员进行培训 测试日程：测试需求分析、测试用例编写、测试实施

  1. 测试设计：测试用例编写、测试场景设计

  2. 测试方法 静态测试：需求评审、设计评审、代码走查、代码检查 黑盒测试：等价划分、边界值、因果图 白盒测试：、逻辑覆盖、语句覆盖、判定覆盖、条件覆盖、判断/条件覆盖、条件组合覆盖

  3. 测试分类 按结构和内部实现 黑盒测试：关心的是输入输出，主要根据规格说明，只依靠被测试程序的输入和输出之间关系或程序的功能来设计测试用例 白盒测试：代码层面 灰盒测试：两者之间

按是否执行程序 静态测试：测试不运行部分，知识检查与审阅 动态测试：运行与使用 按过程：单元测试、集成测试、确认测试、系统测试、验收测试 按测试类型：功能测试、性能测试、安全测试、易用性测试、兼容性测试

  1. 测试执行（量和度的问题） 1、一个版本需要测试哪些方面 2、每个方面要测试到什么程度 3、在有限的时间内，在人员有限甚至短缺的情况下，要考虑如何分工，如何合理地利用资源来开展测试 4、当测试人员测试的执行不到位、敷衍了事时该如何解决 5、测试效率问题 6、根据版本的不同特点是只做验证测试还是采取冒烟测试亦或是系统全面测试 7、当测试过程中遇到一些偶然性随机问题该怎样处理 8、当版本中出现很多新问题时该怎样对待 9、测试停止标准

  2. Bug 生命周期 new：找到新的 Bug renew：重新提交的 Bug open：提交给开发 fixed：开发修复 test：验证是否修复 closed：已修复

  3. Bug 的描述 主题：用一句话简明扼要的描述出软件遇到的问题 环境：主要是描述自己操作软件的环境：比如什么语言的系统，多少位，什么版本，安装的测试软件是什么版本等等 重现步骤：描述 Bug 的重现步骤，在这个描述中药条理清晰，不要添加自己的主观意见，语句中不要出现人称，最好每句话以动词开头，比如：打开什么，点击什么等等，每一步都要简明清晰 实际结果：描述自己看到的实际结果的时候，要客观真实的描述 期望结果：描述期望得到的结果 附加信息：根据实际情况，比如系统、位数
